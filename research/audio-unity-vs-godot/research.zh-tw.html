Unity：音訊系統

  • AudioSource / AudioListener：聲音由物件上的 AudioSource 元件發出，並由 AudioListener（通常掛在 Camera 上）接收——這種來源／聽者的分離架構正是驅動空間音訊模擬的基礎。
  • AudioClip：匯入的音訊資料容器。支援單聲道、立體聲及多聲道；可接受的格式包括 AIFF、WAV、MP3、Ogg，以及 tracker module 格式（.xm、.mod、.it、.s3m）。
  • AudioMixer：用於合併多個音源、套用效果與進行母帶處理的混音圖。
  • 空間／3D 音訊：在 AudioSource 與 AudioListener 之間進行基於距離的衰減與位置模擬；並支援基於兩者相對速度的 Doppler Effect（都卜勒效應）。
  • 回音／殘響的限制：Unity 無法根據場景幾何自動計算回音／殘響——必須透過 Audio Filters 或 Reverb Zones 手動實作。
  • Microphone 類別：透過腳本從可用的麥克風進行即時音訊錄製。

Godot：音訊系統

  • 播放節點依空間維度區分（不同於 Unity 使用單一 AudioSource 加上空間混合設定的方式）：
    • AudioStreamPlayer — 無位置性，輸出至任意音訊匯流排。
    • AudioStreamPlayer2D — 2D 位置性；根據與螢幕邊緣的距離進行立體聲聲像調整；Area2D 可將其輸出重新導向至特定匯流排。
    • AudioStreamPlayer3D — 3D 位置性；依設定可支援立體聲／5.1／7.1 環繞音效；Area3D 同樣可控制其路由。
  • AudioStream 資源：音訊檔案會載入為 AudioStream（及其格式專屬子類別），再指派給播放節點——與 Unity 透過 AudioClip＋AudioSource 所採用的資料／播放器分離概念相同。
  • AudioStreamRandomizer：一種內建資源，可在每次播放時於多個串流間隨機挑選，並可選擇性地變化音高／音量——這是 Godot 原生內建的便利功能，而 Unity 通常得靠手動隨機索引 AudioClip[] 陣列來實現。
  • 透過匯流排實現殘響：AudioStreamPlayer3D 在進入 Area3D 時，可將乾訊號與殘響訊號分別送往不同的匯流排，藉此實現各房間的聲學效果——這是 Godot 對 Unity Reverb Zones 的解法，透過匯流排系統加上 Area3D 實作，而非專屬的區域元件。
  • Doppler Effect：概念上以相同方式支援——追蹤 AudioStreamPlayer3D 與 Camera 之間的相對速度。
  • 音訊匯流排：Godot 的混音圖（Audio 面板），每個匯流排可套用效果——在結構上等同於 Unity 的 AudioMixer 群組。
  • Web 平台的限制：在使用預設的 Sample 播放模式時，殘響匯流排與 Doppler effect 在 Web 匯出平台上不受支援——僅在 Stream 模式下才能運作。
  • 將遊戲玩法與音訊同步（與節奏遊戲／緊密音畫同步相關）：Godot 文件明確記載了兩種做法：
    1. 系統時脈方法（建議用於短音軌）：使用 Time.get_ticks_usec()，並以 AudioServer.get_time_to_next_mix() 與 AudioServer.get_output_latency() 進行補償。長時間播放後會逐漸產生偏移。
    2. 硬體時脈方法（適用於長時間／連續播放）：結合 AudioStreamPlayer.get_playback_position() 與 AudioServer.get_time_since_last_mix()，再透過減去 AudioServer.get_output_latency() 進一步修正。可能因多執行緒而出現輕微抖動——應驗證數值不會逐幀遞減，並捨棄異常值。
    • 文件中提到的延遲來源包括：音訊緩衝區大小（可於專案設定中調整，需在延遲與 CPU 負載／爆音風險間取捨）、以區塊為單位的混音、繪圖 API 延遲（2-3 影格），以及電視／顯示器的處理管線。

回答可能會被問到的問題

1. AudioSource／AudioListener 的直接遷移對應為何？

AudioSource → 依聲音所需的空間性選用對應的 AudioStreamPlayer 變體（無位置性／2D／3D）。AudioListener 這一側仍需驗證——Godot 預設行為（以目前 Camera 作為隱含聽者）或許已能涵蓋大多數情況；請確認此專案是否需要獨立於攝影機、可另行覆寫位置的聽者節點。

2. Unity 的 AudioMixer 如何對應到 Godot？

Godot 的匯流排系統（Audio 面板）在結構上是對應項目——匯流排搭配各匯流排效果，類似 AudioMixer 的群組。請確認此專案實際使用了 AudioMixer 的哪些具體功能（快照、暴露／閃避〔ducking〕參數、旁鏈壓縮），因為本文尚未確認兩者是否能達到 1:1 的功能對等。

3. 緊密的音畫同步（例如節奏遊戲機制）該如何處理？

使用 Godot 文件記載的同步方法：短音軌採用系統時脈方法，長時間／連續播放則採用硬體時脈方法——兩者皆有明確記載的延遲補償公式。若 Unity 端目前使用類似 AudioSettings.dspTime／PlayScheduled() 的取樣精確排程，這部分需要另外對照 Unity 專屬的音訊腳本文件加以確認（本文所查閱的總覽頁面並未涵蓋）。

4. 有沒有平台特定的注意事項？

若此專案的目標平台包含 Web／HTML5 匯出，請注意 Godot 的殘響匯流排路由與 Doppler effect 在 Web 上的預設 Sample 播放模式下無法運作——僅 Stream 模式支援。

後續追蹤事項／待確認事項

  • 確認 Godot 是否有 AudioListener2D／AudioListener3D 節點（或等效機制），可用於覆寫預設以攝影機作為隱含聽者的行為——本次查閱的文件並未確立此點。
  • 確認 Godot 的即時麥克風／音訊輸入錄製 API（可能是 AudioEffectRecord 或類似機制），並與 Unity 的 Microphone 類別進行比較——本次未涵蓋此項目。
  • 盤點此專案實際使用了哪些 AudioMixer 功能（快照、閃避、旁鏈效果）——這將決定 Godot 匯流排系統能滿足需求的完整程度。
  • 確認此專案是否需要節奏遊戲等級的音畫同步；若是，請對照 Unity 目前的實際實作方式（dspTime／PlayScheduled 或其他方式）與 Godot 這兩種文件記載的同步方法。
  • 確認此專案是否以 Web／HTML5 為目標匯出平台，因該平台存在殘響／Doppler 在 Sample 對 Stream 模式上的限制。
