Unity：著色器與 GPU Instancing

  • 什麼是著色器（shader）：一支 GPU 程式。Unity 有三種類別：Graphics Pipeline Shaders（最常見的情況 —— 在一般渲染過程中計算像素顏色）、Compute Shaders（在圖形管線之外進行的一般 GPU 運算），以及 Ray Tracing Shaders。
  • 撰寫方式：ShaderLab（Unity 自有的宣告式語言，包裝實際的 HLSL 程式碼 —— 定義 Properties、SubShader/Pass 結構、Fallback）、Shader Graph（節點式、免程式碼的視覺化編寫方式），或直接手寫 HLSL。
  • Material 的關係：Shader 是程式的容器；Material 參照一個 Shader，並提供實際餵入其中的屬性值（貼圖、顏色、浮點數）。
  • GPU instancing：透過 #pragma multi_compile_instancing 依著色器選擇性啟用。每個實例（per-instance）的屬性會宣告在一個 instancing buffer 中： 
    UNITY_INSTANCING_BUFFER_START(Props)
    UNITY_DEFINE_INSTANCED_PROP(float4, _Color)
UNITY_INSTANCING_BUFFER_END(Props)
    頂點輸入結構（vertex input struct）需要 UNITY_VERTEX_INPUT_INSTANCE_ID；頂點著色器（vertex shader）必須呼叫 UNITY_SETUP_INSTANCE_ID(v)（這是必要的 —— world matrices 依賴它）及 UNITY_TRANSFER_INSTANCE_ID(v, o) 以將 ID 傳遞到片段（fragment）階段；片段著色器則透過 UNITY_ACCESS_INSTANCED_PROP(Props, _Color) 讀取數值。
    • 最小可行的 instanced shader（Built-in pipeline，ShaderLab/HLSL）： 
      Shader "Custom/InstancedColor"
{
    Properties { _Color ("Color", Color) = (1,1,1,1) }
    SubShader { Pass {
        CGPROGRAM
        #pragma vertex vert
        #pragma fragment frag
        #pragma multi_compile_instancing
        #include "UnityCG.cginc"

        UNITY_INSTANCING_BUFFER_START(Props)
            UNITY_DEFINE_INSTANCED_PROP(float4, _Color)
        UNITY_INSTANCING_BUFFER_END(Props)

        struct appdata { float4 vertex : POSITION; UNITY_VERTEX_INPUT_INSTANCE_ID };
        struct v2f { float4 pos : SV_POSITION; UNITY_VERTEX_INPUT_INSTANCE_ID };

        v2f vert(appdata v) {
            v2f o;
            UNITY_SETUP_INSTANCE_ID(v);
            UNITY_TRANSFER_INSTANCE_ID(v, o);
            o.pos = UnityObjectToClipPos(v.vertex);
            return o;
        }
        fixed4 frag(v2f i) : SV_Target {
            UNITY_SETUP_INSTANCE_ID(i);
            return UNITY_ACCESS_INSTANCED_PROP(Props, _Color);
        }
        ENDCG
    } }
}
    • 一旦著色器支援此功能，且在 material 上勾選了「Enable GPU Instancing」，共用同一組 mesh + material 的一般 GameObject 就會被 Unity 的渲染器自動批次處理（batched）成較少的 draw call —— 不需要另外的、專屬於 instancing 的節點／物件類型。每個實例的數值（如上面的 _Color）通常是透過每個 renderer 的 MaterialPropertyBlock 來設定。

Godot：著色器與 MultiMesh

  • 什麼是著色器（shader）：一個以 Godot 自有著色語言（.gdshader，類似 GLSL 的語法、Godot 專屬的內建函式）撰寫的 Shader 資源。shader_type 宣告其目標對象：spatial（3D）、canvas_item（2D）、particles、sky 或 fog。
  • Material 的關係：ShaderMaterial 參照一個 Shader，並透過 set_shader_parameter()／get_shader_parameter() 提供 uniform 數值 —— 程式碼中的 uniform 名稱必須完全相符。
  • GPU instancing：這不是一個著色器旗標（flag），而是一個獨立的資源／節點類型：MultiMesh（透過 MultiMeshInstance3D／MultiMeshInstance2D 使用）。設定 instance_count，然後針對每個實例：set_instance_transform()、set_instance_color()（需要 use_colors = true），以及 set_instance_custom_data()（需要 use_custom_data = true；資料會被打包成一個 Color，也就是每個實例 4 個浮點數）。著色器則透過頂點階段內建的 INSTANCE_CUSTOM 讀取這些資料。
    • 最小可行的 instanced shader（.gdshader）： 
      shader_type spatial;

void vertex() {
    // INSTANCE_CUSTOM.rgb holds up to 3 custom floats set via set_instance_custom_data()
    COLOR = vec4(INSTANCE_CUSTOM.rgb, 1.0);
}

void fragment() {
    ALBEDO = COLOR.rgb;
}
    • 設定方式（GDScript）： 
      var multimesh := MultiMesh.new()
multimesh.mesh = my_mesh
multimesh.use_custom_data = true
multimesh.instance_count = 1000

for i in range(1000):
    var t := Transform3D(Basis(), Vector3(i * 2.0, 0, 0))
    multimesh.set_instance_transform(i, t)
    multimesh.set_instance_custom_data(i, Color(randf(), randf(), randf(), 1.0))

$MultiMeshInstance3D.multimesh = multimesh
    • 文件中提到的注意事項：每個實例最多只有 4 個自訂浮點數（這是實際的容量限制）；一旦某個 MultiMesh 批次共用的光照預算（light budget）用盡，其餘的實例就不會再獲得進一步的照明；整個批次會被當作單一物件進行空間索引（distant instances 可能仍會被渲染，因而損及裁剪／culling 效果）；blend shapes 會被忽略。

比較／對應關係

概念UnityGodot
著色器語言 透過 ShaderLab 包裝的 HLSL，或視覺化的 Shader Graph Godot Shading Language（.gdshader，類似 GLSL）
管線目標設定 由目前使用中的渲染管線（Built-in/URP/HDRP）+ Pass 結構決定 透過 shader_type 明確指定：spatial/canvas_item/particles/sky/fog
Material 與 Shader 的連結 Material → Shader，透過 Inspector／MaterialPropertyBlock 設值 ShaderMaterial → Shader，透過 set_shader_parameter() 設值
Instancing 的啟用方式 依著色器選擇性啟用的旗標（#pragma multi_compile_instancing）；一般 GameObject 會自動批次處理 明確使用 MultiMesh 資源／節點 —— 這是完全不同的建構方式，而非一般節點上的旗標
每個實例的資料容量 型別化的 instancing buffer，主要受限於 GPU constant-buffer 大小（相對寬裕） 固定：transform + color + 4 個浮點數的自訂資料（INSTANCE_CUSTOM）—— 一個明確且較小的上限
實例是否為完整的場景物件？ 是 —— 每個實例仍是帶有元件（component）的一般 GameObject 否 —— MultiMesh 的實例純粹是渲染資料（緩衝區中的索引），而非 Node；沒有每個實例各自的腳本／物理／碰撞

兩個引擎為何如此設計

  • Unity：由於 Unity 的渲染器本來就把每個 Renderer 視為在共同管線中流動的獨立物件，GPU instancing 是以管線最佳化的方式疊加上去的：宣告著色器支援、在 material 上啟用，引擎便會在渲染時透明地將同 mesh／同 material 的 GameObject 批次處理。這讓場景編寫模型維持不變（仍是一般的 GameObject）—— 代價是每個著色器要設定 instancing 相當繁瑣（需要多個巨集），而且如果著色器變體（shader variant）或 material 不匹配，批次處理可能會悄悄地無法觸發。
  • Godot：MultiMesh 是一個刻意獨立、明確的資源 —— 這與 Godot 一貫「用多少、付多少」（pay for what you use）的設計模式一致。選擇使用 instancing，代表你選擇進入一種真正不同的表示方式（資料存在緩衝區中，而非 Node），以此換取每個實例場景圖（scene-graph）便利性（沒有各自的訊號／物理／腳本）的犧牲，來獲得非常可預測、明確的效能。這也呼應了先前 SRP/URP vs RenderingDevice/Compositor 的研究發現：Godot 並不像 Unity 的 SRP 那樣支援可替換的自訂渲染管線，因此它也不需要類似 ShaderLab 那樣的間接層（indirection layer）——只要一種著色語言、直接綁定在引擎固定的渲染器上，就已足夠。

關鍵遷移風險

如果 Unity 專案中使用 GPU instancing 的物件同時也帶有各自的遊戲邏輯（碰撞體、腳本、每個 GameObject 各自的行為），直接搬移到 MultiMesh 會遺失這些東西 —— MultiMesh 的實例沒有各自的 Node 身分，因此沒有東西可以讓你在每個實例上掛載碰撞體／腳本。在這種情況下，正確的遷移方式是採用混合式（hybrid）做法：保留一個輕量的外部資料模型（例如一組 struct／Resource 陣列）來驅動遊戲邏輯，並另外將 transform／自訂資料每一影格或在變更時推送到 MultiMesh，純粹用於渲染。

後續追蹤事項／待日後確認

  • 確認專案中使用 instancing 的 GameObject 是否帶有各自的遊戲邏輯元件（碰撞體／腳本）—— 這將決定是否需要如上述遷移風險中所說的「混合式資料模型 + MultiMesh」做法。
  • 盤點專案中使用 instancing 的著色器，除了 transform／color 之外實際用了多少每個實例的自訂資料 —— Godot 的 4 個浮點數上限，若需求更多，可能需要打包技巧（packing tricks）或以 instance ID 索引的獨立每實例資料貼圖（per-instance data texture）。
  • 確認 Unity 專案中使用 instancing 的著色器是針對哪個渲染管線（Built-in/URP/HDRP）—— 這會影響確切的 HLSL 語法／include，也與 SRP/URP research entry 相關聯。
  • 檢查 Unity 專案是否有使用 Shader Graph —— Godot 的視覺化著色器編輯器（VisualShader）是較接近的編寫工具對應物，若大量使用的話，可能值得建立獨立的研究項目。
