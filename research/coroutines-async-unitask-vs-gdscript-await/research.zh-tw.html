Unity：三種並存的非同步機制

  1. Coroutine（IEnumerator + StartCoroutine／StopCoroutine）
    • 函式會透過 yield return 暫停並在之後恢復執行，這是整合進 Unity 影格迴圈（frame loop）中的機制——並非真正的執行緒（thread）。
    • yield return null → 於下一個影格恢復；yield return new WaitForSeconds(n) → 延遲一段時間後恢復；其他 YieldInstruction 涵蓋更多情況；Coroutine 可以巢狀（nest）（yield return StartCoroutine(...)）。
    • 無法直接回傳數值（沒有泛型結果）——需以 ref／callback 模式來變通處理。
    • 自動綁定於所屬 MonoBehaviour 的生命週期（若 GameObject 被停用／銷毀，Coroutine 會停止）。
  2. 原生 async/await（Awaitable，於 Unity 2023.1+ 引入，與標準 .NET Task 並存）
    • 真正的 C# async／await；與 Coroutine 不同，它可以回傳數值。
    • 新增 Awaitable 的原因，正是因為單純的 Task 帶有 GC 配置（allocation）的額外負擔，且不符合 Unity 單一主執行緒（single-main-thread）的執行模型。
    • 需要留意依據 continuation 恢復執行時所在的執行緒，哪些 Unity API 可以安全呼叫。
  3. UniTask（Cysharp，第三方，免費）
    • 以 struct 為基礎、零配置（zero-allocation）的 UniTask<T> 解決 Task 的配置額外負擔問題。
    • 整合 Unity 的 PlayerLoop 以進行以影格為單位的等待（UniTask.Yield()、UniTask.DelayFrame()），無需啟動真正的執行緒。
    • 採用標準的以 CancellationToken 為基礎的取消機制。
    • 支援 AsyncEnumerable／LINQ 用於非同步串流（async streams），以及事件／UI 綁定。
    • 注意事項：除非呼叫 .Preserve()，否則一個 UniTask 實例只能被 await 一次——這相較於 Task（可被多次 await）以及 Coroutine（概念上沒有這種限制）而言，是個實際存在的陷阱（footgun）。
    • 預設不使用多執行緒——必須明確呼叫 UniTask.SwitchToThreadPool() 才能離開主執行緒。

Godot：await（GDScript，Godot 4）

  • GDScript 只有一種統一的非同步模型，而非三種：任何包含 await 的函式，都會隱含地成為一個可被其呼叫端以 await 使用的 coroutine。
  • await 會暫停執行，直到某個signal觸發（或被 await 的呼叫完成）——例如 await get_tree().create_timer(1.0).timeout，或是 await 一個自訂／節點的 signal。
  • 在沒有 await 的情況下呼叫一個類似 coroutine 的函式，並不會造成阻塞（block）——它會「立即回傳一個可能被忽略的值」，而被呼叫端則繼續在背景執行（即發即忘，fire-and-forget）。
  • 在同一個影格內發出的自訂 signal，應使用 call_deferred 以避免時機／順序問題。
  • Godot 3.x 使用的是 yield() 而非 await，它會回傳一個 GDScriptFunctionState 物件，並需手動呼叫 .resume() 方法——相較於 Godot 4 的 await，這是一種更手動、更低階的機制。

比較／對應關係

Unity Godot 備註
Coroutine（IEnumerator、StartCoroutine） 對某個 signal 使用 await（函式隱含地成為 coroutine） Godot 沒有獨立的「coroutine」API 介面——它就只是 await 加上 signal，與一般函式統一在一起
yield return null（等待一個影格） await get_tree().process_frame （請確認確切的 signal 名稱／時機——見後續追蹤事項）
yield return new WaitForSeconds(n) await get_tree().create_timer(n).timeout
StopCoroutine() （無直接對應項目） 需要手動的取消模式；在已釋放（freed）的節點上恢復 await 是常見的執行期錯誤
原生 async／Awaitable（Unity 2023.1+） await（GDScript 原生關鍵字） 兩者皆為「真正的」async/await，且與 Coroutine 不同，兩者都能回傳數值
UniTask（第三方、零配置、與 PlayerLoop 整合） （不需要第三方層） GDScript 的 await 本身已是原生、與影格整合的語言特性——它所解決的問題，正是 UniTask 為 Unity 事後補上的同一個問題
CancellationToken（Task/UniTask） （無內建對應項目） 需要手動的旗標／保護（guard）模式
UniTask.SwitchToThreadPool() Thread / WorkerThreadPool 真正的多執行緒在兩個引擎中都是各自獨立、需選擇啟用（opt-in）的系統，並非 await/coroutine 模型本身的一部分

回答可能會被問到的問題

1. GDScript 有對應 Unity Coroutine 的東西嗎？

沒有獨立的 API——await 加上 signal 已原生涵蓋相同的範疇。任何使用 await 的函式都會自動表現得像 Unity 所稱的 coroutine，不需要學習另一套類似 IEnumerator 的機制。

2. GDScript 有類似 C# 原生 async/Awaitable 的東西嗎？

有，而且它是 GDScript 中唯一的非同步模型——沒有需要額外考量的舊有替代方案。相較之下，Unity 有三套並存的系統（Coroutine、原生 async/Awaitable，以及 UniTask），依使用位置的不同，每一套可能都需要不同的遷移處理方式。

3. 我們需要在 Godot 中使用類似 UniTask 的函式庫嗎？

不需要。await 已經是原生的語言特性——預設為單執行緒，並透過 signal 與影格整合——因此它並不存在 UniTask 為了解決 Unity 的 Task 所要克服的那些配置（allocation）／PlayerLoop 整合問題。

4. 最大的遷移風險是什麼？

會失去 Unity 那種在物件銷毀時自動停止 coroutine 的行為。Unity 的 Coroutine 會在其所屬的 MonoBehaviour 被停用／銷毀時自動停止；GDScript 的 await 沒有自動對應的機制，而在已釋放的節點上恢復 await 是已知的執行期錯誤。任何依賴這種自動停止行為的邏輯，都需要在 Godot 移植版本中寫入明確的保護機制（例如在 await 前後檢查 is_instance_valid()）。

後續追蹤／待日後確認的事項

  • 清點專案中每個呼叫點所使用的是 Unity 三種非同步機制（Coroutine／原生 async-Awaitable／UniTask）中的哪一種——正確的 Godot 端轉譯方式會依機制而有所不同。
  • 確認 Unity 專案的版本：Awaitable 僅存在於 Unity 2023.1+；較舊的專案可能只有 Coroutine 加上 UniTask，而沒有原生的 Awaitable 可供比較。
  • 在依賴 Godot 每影格的 await signal（process_frame 相對於物理影格的對應項目）來取代 WaitForEndOfFrame／WaitForFixedUpdate 模式之前，先對照目前 Godot 版本的官方文件確認其確切名稱／時機——此輪未獨立驗證。
  • 檢查是否有透過 ref 參數或 callback 來偽造「回傳值」的 coroutine 程式碼——由於 GDScript 的 await 原生支援真正的回傳值，這種模式很可能可以簡化，而非逐字移植。
  • 在遷移任何目前依賴 StopCoroutine() 或隱含的銷毀即停止語意的 coroutine 之前，先為 GDScript 設計好取消／銷毀保護（destroy-guard）模式。
