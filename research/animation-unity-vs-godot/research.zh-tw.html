Part A —— 角色/屬性動畫

Unity：Mecanim（現代系統）

  • AnimationClip：動作的基本單位（單一的「Idle」/「Walk」/「Run」等），可從 DCC 工具匯入（FBX）或直接在 Unity 的 Animation 視窗中製作。支援 Curves（可隨時間驅動任意屬性，甚至是 Animator 參數）、Animation Events（在播放的特定時間點呼叫指令碼方法）、遮罩（限制於特定骨骼）以及 loop optimization（對齊起始/結束姿勢以達成無縫循環）。
  • Animator 元件：附加於 GameObject 上；參照一個 Controller，以及（對於人形角色）一個 Avatar。主要欄位：Apply Root Motion、Update Mode（Normal / Animate Physics / Unscaled Time）、Culling Mode。
  • Animator Controller：包含狀態機、混合樹（blend trees）與參數的資產，在 Animator 視窗中以視覺化方式編輯。
    • 狀態機：狀態（各自持有一個 Motion——一個 clip 或一個 blend tree）加上狀態之間的轉場（transitions）。特殊節點：Entry/Exit、Any State（不論目前處於哪個狀態都能轉入任何狀態——用於「Hit」/「Death」這類全域中斷），以及用於巢狀結構的 子狀態機。
    • 轉場（Transitions）：混合時長（blend duration）加上條件；同時提供 Exit Time、Transition Offset 與 Interruption Source（進行中的轉場本身是否可被中斷）。
    • 參數（Parameters）：Float、Int、Bool、Trigger，透過 Animator.SetFloat/SetInteger/SetBool/SetTrigger 設定，並由轉場條件與 blend trees 讀取。
    • Blend Trees：依參數值連續混合多個 clips（clips 應具有相似的時間軸/normalized-time 對齊）。類型：1D（單一 float 參數，例如依 Speed 從 Walk 到 Run）、2D（兩個參數；子模式有 Simple Directional / Freeform Directional / Freeform Cartesian），以及 Direct（每個 Motion 的權重各自由其自己的參數驅動）。
    • Animation Layers：每個 Controller 可平行執行多個狀態機。每一層都有一個 Weight、一種混合模式（Override 或 Additive——Additive 需要 additive clip 與基礎層之間的屬性相符）、一個 Avatar Mask（限制於特定骨骼，例如僅上半身）、並可設為 Synced Layer（沿用另一層的狀態機結構但套用不同的 clips，並附帶一個 Timing 選項，用於選擇拉伸至來源長度或依混合長度）。
  • Avatar 系統（Humanoid vs Generic）：Humanoid avatars 會將模型的骨骼對應到一個標準化骨架，使 clip 能在不同人形模型之間重定向（retargeting）（在一個角色上製作的 walk clip 可以驅動比例不同的另一個角色）。需要骨骼對應（手動，或透過 Human Template 預設）、肌肉定義（關節限制），以及一個大致呈 T-pose 的來源模型。Generic avatars 適用於非人形骨架，且無法在不同骨架之間重定向。
  • Root Motion：角色的實際位置/旋轉是否由動畫資料驅動。Body Transform（質心）vs Root Transform（Y 平面投影，逐幀計算）；每個 clip 的烘焙控制項（Root Transform Rotation、Position (Y)、Position (XZ)，各自帶有「Bake Into Pose」加上一個「Based Upon」設定）。啟用時，Unity 會在每幀對該物件的指令碼呼叫 OnAnimatorMove()，以便在套用之前攔截/調和 root-motion 差量與物理或 NavMeshAgent 的關係——若設定不一致，這常是造成錯誤（腳滑、位置跳動）的來源。
  • 輔助工具：Animator Override Controller（沿用某個 controller 的結構但套用不同的 clips）、Playables API（比 Controller 圖表更底層的執行期動畫圖表 API，用於超出 Controller 圖表範圍的程式化動畫圖表）。

Unity：Legacy 動畫系統

  • 元件：Animation（與 Animator 不同）。欄位：預設的 Animation clip、Animations 陣列、Play Automatically、Animate Physics、Culling Type。
  • 沒有狀態機、沒有 Controller 資產、沒有參數、沒有 blend trees——只是一份透過指令碼命令式（imperatively）控制的扁平 clip 清單：Animation.Play()、Animation.CrossFade()（搭配 PlayMode.StopSameLayer/StopAll），以及公開手動混合權重、層（layers）、wrap mode 與 AnimationBlendMode.Additive 的 AnimationState 物件。
  • 用於 Legacy 的 clips 必須在匯入時設定 Legacy 旗標——現代的 Mecanim clips 除非標記，否則無法在此使用。
  • 現況：仍受支援，但精神上已明確被棄用——Unity 自己的文件寫著「對於新專案，請使用 Animator 元件」。保留給舊專案，或用於真正瑣碎的情境（例如道具上單一循環的 clip），此時其較低的開銷仍有意義。
面向Legacy（Animation）Mecanim（Animator）
控制模型命令式（指令碼呼叫 Play/CrossFade）宣告式狀態機 + 參數
混合手動權重／CrossFade 呼叫內建轉場、blend trees、layers
重定向（Retargeting）無Avatar 系統（人形重定向）
Root motion原生不支援一級功能
Clip 需求必須標記為「Legacy」標準 AnimationClip
編輯器工具無Animator 視窗（視覺化圖表編輯器）

Godot：AnimationPlayer

  • 一個 Node（非 spatial），作為一個或多個 Animation 資源的資料容器，並附有時間軸/關鍵影格編輯器。重要注意事項：由於它只是一個普通的 Node，直接掛在 AnimationPlayer 底下的 2D/3D 子節點不會繼承其變換（transform）——不要把場景內容直接掛在 AnimationPlayer 底下（這是從 Unity 帶過來的常見習慣，Unity 中 Animator 是直接掛在它所驅動的 GameObject 上）。
  • Animation Libraries：個別的 Animation 資源會被登錄到 Animation Library 資源中，以便在不同的 AnimationPlayer 之間重複使用/共享。
  • 軌道（Tracks）：時間軸上的一行，綁定到一個節點路徑加上一個屬性，於第一個關鍵影格建立時自動產生。每條軌道有：Update Mode（Continuous / Discrete / Capture）、Interpolation（Nearest / Linear / Cubic / Linear Angle / Cubic Angle——Angle 變體會走最短的角度路徑，這對旋轉很重要）、Loop Mode（Clamp / Wrap）。
  • 標記（Markers）：時間軸上的具名節點，用於透過 play_section_with_markers() 播放某個子區段。
  • RESET 動畫：一個特殊的 _RESET 動畫，儲存場景的預設姿勢；同時用於場景儲存一致性（「Reset On Save」）以及作為 AnimationTree 混合時，動畫未涵蓋屬性所使用的預設回退值。
  • 混合（Blending）：clips 之間簡單、固定時長的交叉淡化（crossfade）——這正是促成 AnimationTree 存在的動機性限制。

軌道類型（來自 animation_track_types.html）：

軌道類型用途備註
Property Track基礎軌道——可動畫化任何暴露的屬性所有專門類型都是它的最佳化變體
Position/Rotation/Scale 3D專用的變換軌道支援壓縮；用於匯入的 3D 模型動畫
Blend Shape Track動畫化 MeshInstance3D 的 morph targets同樣可壓縮
Method Call Track在指定時間點以參數呼叫方法在編輯器預覽期間會被抑制（安全考量）——直接對應 Unity 的 Animation Events
Bezier Curve Track沿手動編輯的曲線動畫化單一 float/屬性在同一混合情境下不能與 property tracks 混用——這是硬性限制
Audio Playback Track在時間軸上放置音訊 clips支援依混合權重調整音量
Animation Playback Track觸發其他 AnimationPlayer 的動畫可編排多節點/過場動畫的時間軸

Godot：AnimationTree

  • 其存在專門是為了解決 AnimationPlayer「僅能固定時長交叉淡化」的限制。不擁有動畫資料——它參照並驅動一個既有的 AnimationPlayer（在 Player 中製作 clips，在 Tree 中接上混合/狀態邏輯）。概念上，它把 Unity 打包在一起的 Animator（元件）+ Controller（資產）拆分成兩個節點。
  • 根節點類型：AnimationNodeAnimation（單一 clip）、AnimationNodeBlendTree（由 Blend2/Blend3/OneShot/TimeSeek/TimeScale/Transition 節點組成的圖表）、AnimationNodeBlendSpace1D/BlendSpace2D（在一條線/一個平面上對動畫點做參數化插值）、AnimationNodeStateMachine（具名狀態加轉場；可在狀態內巢狀 blend trees/spaces/其他狀態機）。
  • 狀態機轉場：切換模式有 Immediate / Sync（將新狀態快轉以對齊舊狀態的位置）/ At End；屬性有 Xfade Time、Xfade Curve、Reset、Priority（供尋路使用）、Advance Mode。Advance Condition/Expression 會依布林參數或任意表達式（例如 is_walking && !is_idle）自動觸發轉場。travel() 可在狀態圖上尋路： 
    var state_machine = animation_tree["parameters/playback"]
state_machine.travel("TargetState")
  • Blend tree 節點：Blend2/Blend3（2 至 3 個輸入的加權混合，附帶軌道過濾功能以支援分層——例如僅限上半身的疊加，這是 Godot 對 Unity Avatar Mask + Additive layers 的回應）、OneShot（播放一次後還原）、TimeSeek、TimeScale、Transition（輕量具名輸入切換器）。範例： 
    animation_tree["parameters/OneShot/request"] = AnimationNodeOneShot.ONE_SHOT_REQUEST_FIRE
  • Blend spaces：BlendSpace2D 會在 2D 平面上的動畫點之間自動三角化（Delaunay）並混合最鄰近的點——直接對應 Unity 的 2D Blend Tree。Sync modes（None/Independent/Cyclic Mutable/Cyclic Constant）控制非主導的點是否仍會推進時間；cyclic 模式要求每個混合點都必須是長度有限、不可變的普通 AnimationNodeAnimation（在這些模式下不能將子樹作為點來巢狀使用）。
  • 混合正確性要求：每個被動畫化的屬性都需要一個確定的靜止值（Skeleton3D 骨骼使用 Bone Rest；其他屬性預設為 0 或某個 RESET 動畫的第一個關鍵影格）。在給定動畫中未被追蹤的屬性，在混合期間會被視為處於靜止狀態。人形骨架應以 T-pose 匯入，使骨骼靜止姿勢處於中間範圍——這與 Unity 自身對 Avatar 對應所要求的 T-pose 直接對應。在混合動畫之間需要超過 180° 旋轉的骨骼，應改用 root motion，因為插值無法消解超過 180° 的旋轉方向歧義。
  • Root Motion：指定某根骨骼作為 Root Motion Track；Godot 在正常播放期間會取消該骨骼的視覺變換。執行期 API 是主動拉取差量，而非 Unity 的推送式回呼模型： 
    animation_tree.get_root_motion_position()
animation_tree.get_root_motion_position_accumulator()  # cumulative, for blended cases
    再餵入 CharacterBody3D.move_and_slide()。概念上與 Unity 的 OnAnimatorMove() 模式相對應，但 Godot 的做法是明確的逐幀拉取，而不是引擎主動呼叫的回呼。
  • 共用資源注意事項：AnimationTree 內部的圖表節點是資源（resources）——如果同一個 AnimationTree 資源在多個場景實例間共享，修改其中一個實例的節點設定可能會外洩到全部實例。
  • 參數（Parameters）：透過屬性路徑暴露（animation_tree["parameters/eye_blend/blend_amount"] = 1.0），可在編輯器中檢視（Advance Expressions 除外，它們是以表達式文字存在，而非簡單的參數）。

Godot：2D 骨架與 Cutout Animation（IK）

  • 2D 骨架：透過權重繪製，以 Skeleton2D/Bone2D 階層來形變一個 Polygon2D 網格。此頁面不含 IK——2D IK 位於 Cutout Animation 之中。
  • Cutout Animation：基於精靈圖的「人偶」綁定——透過樞轉的精靈圖階層、以「Make Bones」在精靈節點上建立骨架，以及用於藉由拖曳末端執行器（end-effector）擺姿勢的 2D IK 鏈。重要落差：2D IK 鏈僅限編輯器內使用，用於製作關鍵影格——這套系統中沒有執行期 IK 求解，這與 Unity 的 Animation Rigging 套件不同（該套件是獨立套件，尚未確認本專案是否使用）。

Part B —— 影片播放與電影/遊戲錄製

Unity：VideoPlayer

  • 將影片播放到 GameObject 的貼圖、Render Texture、相機平面，或目標顯示裝置上的元件，來源可以是 VideoClip 資產（由資產管線管理）或 URL（file:///http://，繞過資產管線——執行期的可連線性由你自行負責）。
  • Play() 會在需要時自動準備（造成延遲）；若要立即播放，可提前呼叫 Prepare() 並等待 prepareCompleted。支援全景/360 度影片與播放同步 API。
  • 編碼支援範圍廣泛，通常取決於平台/硬體解碼器（一般為 MP4/H.264 家族）——取代了較舊、已棄用的 MovieTexture 系統。

Unity：螢幕/遊戲錄製（Unity Recorder）

  • Unity 核心中沒有內建的執行期錄製 API。 Unity Recorder 套件（com.unity.recorder，透過 Package Manager 安裝）是標準工具：Movie Recorder（H.264 MP4、VP9 WebM、ProRes QuickTime）、Image Sequence、Animation Clip、GIF 及 Audio 錄製器。
  • 僅限編輯器使用——在編輯器的 Play 模式下可運作，但在獨立建置版本中無法運作。不能用於出貨後的遊戲內/玩家端錄製。

Godot：VideoStreamPlayer

  • 用於影片播放的節點。格式支援範圍狹窄且是刻意為之：實際上僅支援 Ogg 容器（偏好 .ogv）中的 Ogg Theora 影片（+ 選用的 Ogg Vorbis 音訊）。基於軟體專利原因排除了 H.264/H.265（這是授權層面的決策，而非技術限制）；WebM 支援已在 Godot 4.0 中移除（維護負擔考量）；因缺乏廣泛的 GPU 解碼支援而避開 AV1。Theora 解碼僅限 CPU，限制了實際可用的解析度（桌面版大約 1080p30，行動裝置/網頁版大約 720p30 或更低，作為粗略上限）。
  • 不支援 URL 串流——檔案必須是本機/已匯入的。音訊最多下混為立體聲。
  • 遷移影響：既有的 MP4/H.264 影片資產需要轉碼為 Ogg Theora，相較於 Unity 通常採用硬體解碼的格式，畫質/效能上限預期會下降。

Godot：Movie Maker 模式

  • 一個非即時、確定性的逐影格渲染器，輸出到影片檔案——明確地說不是通用的遊戲畫面錄製工具（文件中建議改用 OBS Studio/SimpleScreenRecorder 來滿足該用途）。用途：預告片、高品質預先渲染的過場動畫，以及在負載下不受掉影格/音訊不同步問題困擾的確定性擷取，而這些問題常見於即時錄製。
  • 透過編輯器的膠捲圖示啟用（每次工作階段都需要重新啟用），或透過 CLI：godot --path /path/to/project --write-movie output.avi（也適用於已匯出的建置版本，只是無法在執行期切換）。
  • 輸出格式：OGV（Theora+Vorbis，最佳的檔案大小/畫質比，僅限編輯器建置版本，無法在瀏覽器中播放）、AVI（MJPEG+PCM，編碼速度快，有 4 GB 上限）、PNG 序列 + WAV（無損，用於外部合成，支援透明度）。
  • 必須乾淨地結束（關閉視窗／呼叫 get_tree().quit()／使用 AnimationPlayer 的「Movie Quit On Finish」）——強制關閉會損毀 AVI/WAV 的檔頭。提供 OS.has_feature("movie")，讓指令碼能在擷取期間切換成即時渲染時太慢、只有擷取時才可用的高品質設定。由於擷取速度不受即時限制約束，因此支援超取樣（supersampling）。

回答可能會被問到的問題

1. 典型 Mecanim 設定的直接遷移對應是什麼？
AnimationClip → Animation 資源；Animator+Controller → AnimationPlayer+AnimationTree；Controller 的狀態機 → AnimationNodeStateMachine；blend trees → AnimationNodeBlendSpace1D/2D 或 Blend2/Blend3；參數 → AnimationTree 的 parameters/... 路徑。

2. 那仍在使用 Legacy Animation 元件的專案呢？
在 Godot 中沒有獨立的「legacy」概念需要保留——一個不搭配 AnimationTree 的普通 AnimationPlayer 就已經涵蓋了簡單、無混合、由指令碼驅動的播放，這正是 Legacy 過去的用途。在這方面的遷移，會比 Mecanim 更豐富的功能集要來得直接許多。

3. Godot 是否支援像 Unity Avatar 系統那樣的人形重定向？
此研究階段尚未確認——這需要一個專門的後續追蹤，因為如果本專案在多個比例不同的人形角色之間共享動畫 clips，這會是一項重要功能。

4. Root motion 如何延續過去？
概念上是直接對應的——兩個引擎都支援由動畫驅動的角色移動——但控制流方向相反：Unity 每一幀透過 OnAnimatorMove() 將差量推送給你的指令碼；Godot 則要求你自行透過 get_root_motion_*_accumulator() 拉取差量，再餵入 CharacterBody3D.move_and_slide()。

5. 影片資產需要做哪些變更？
任何既有的 MP4/H.264（或 WebM）影片都需要轉碼為 Ogg Theora（.ogv）——Godot 的 VideoStreamPlayer 不支援 H.264/H.265（專利原因）或 WebM（4.0 中已移除），而且 Theora 解碼僅限 CPU，相較於 Unity 通常採用硬體解碼的播放方式，這會限制實際可用的解析度/影格率。

6. 兩邊是否都有遊戲內錄製功能？
兩邊都沒有。Unity Recorder 僅限編輯器使用（無法用於建置版本）；Godot 的 Movie Maker 模式是確定性、非即時的擷取工具，同樣不是即時的遊戲錄製工具。如果專案需要面向玩家的錄製功能，不論使用哪個引擎，都需要自訂或第三方解決方案。

後續追蹤事項／待確認事項

  • 確認本專案實際使用哪一種（或哪幾種）Unity 動畫系統——僅 Mecanim、僅 Legacy，或是混合使用——因為這會改變 Legacy 專屬遷移指引的適用程度。
  • 如果專案依賴 Unity 基於 Avatar 的 clip 重定向、在多個角色模型之間共用動畫，則需要專門研究 Godot 4.x 的人形重定向/骨骼對應能力（本輪尚未涵蓋）。
  • 確認是否使用了搭配 Avatar Masks 的 Animation Layers（例如僅限上半身的疊加），並驗證 Godot 的 Blend2/Blend3 軌道過濾功能是否能涵蓋相同的細緻程度。
  • 確認是否使用了 Unity 的 Synced Layers 功能（沿用某一層的狀態機結構但套用不同 clips）——本輪未找到已確認的 Godot 對應功能。
  • 確認是否有任何地方需要執行期 2D IK——Godot 的 Cutout Animation IK 僅限編輯器使用（製作關鍵影格的輔助工具），並非執行期求解器，若專案在執行期依賴 Unity 的 Animation Rigging 套件，這會是一個真正的落差。
  • 具體確認 root motion 的使用情況，包括任何基於 OnAnimatorMove、與 NavMeshAgent 或物理系統調和的邏輯，因為這部分邏輯需要圍繞 Godot 的拉取式累加器 API 重新架構。
  • 盤點既有影片資產的編碼/解析度——這會決定 Theora 轉碼的範圍，以及 CPU 解碼的解析度上限對本專案的使用情境而言是否可接受。
  • 確認專案是否需要遊戲內/面向玩家的影片或遊戲錄製功能——因為兩個引擎都未內建此功能，不論選哪個引擎都需要同類型的自訂/第三方解決方案（這不是遷移特有的落差）。
  • 確認是否使用了 Unity 的 Playables API 來實現超出 Animator Controller 所提供範圍的程式化動畫圖表——本輪未找到 Godot 對應功能；若有依賴，需要另外研究。
