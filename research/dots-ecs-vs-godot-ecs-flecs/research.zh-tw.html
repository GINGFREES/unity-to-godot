Unity：DOTS/ECS

  • DOTS（Data-Oriented Technology Stack）是三個協同組件的統稱：Entities package（ECS 架構本身）、C# Job System，以及 Burst Compiler。這三者的設計目的是彼此加乘，而非獨立使用。
  • 核心概念：Entity（輕量級 ID）、Component（IComponentData —— 純資料、無邏輯）、System（SystemBase，受管理，或 ISystem，一種不受管理、相容 Burst 的 struct —— 也就是對符合查詢條件的 entities 進行操作的邏輯）、World（管理 entities 與 systems 的容器）。
  • Archetype/chunk 記憶體佈局：擁有相同 component 組合的 entities 屬於同一個 Archetype；記憶體以 64KB chunks 配置，每個 chunk 存放多個相同 archetype 的 entities，其 components 以線性、連續的方式儲存——這是一種對快取友善、類似 Structure-of-Arrays 的佈局，刻意與個別 GameObject/MonoBehaviour 資料分散於記憶體中的方式不同。
  • C# Job System：將 Unity 內部的 C++ job system 暴露給 C# 使用，讓 systems 中逐 entity 的工作（IJobEntity/IJobChunk）可以安全地排程到多個 worker 執行緒上——這裡的「安全」是指 Unity 會靜態追蹤每個 job 的讀寫存取，避免衝突的 jobs 產生 race condition，而彼此獨立的 jobs 則會自動並行執行。
  • Burst Compiler：一個基於 LLVM 的編譯器，能將受限、「相容 Burst」的 C# 子集（不含受管理的參考、不含 GC 配置）直接轉譯為最佳化的原生機器碼——這正是讓 DOTS 程式碼在不脫離 C# 語法的情況下，達到接近原生速度的關鍵。
  • 整體效果：ECS 將資料組織成適合大量、快取友善處理的形式；Jobs 將該處理平行化到多個核心；Burst 將熱路徑上逐 entity 的程式碼編譯為原生速度。需要 Unity 2022.3.0f1 以上版本。Unity 官方的指引是將 ECS 視為與一般 GameObjects/MonoBehaviours 共存，用於真正需要這種規模的子系統，而非全面取代。

Godot：沒有官方 ECS——三種第三方策略

Godot 並未內建 ECS；預設架構是 Nodes + scripts（GDScript 或 C#）。所給出的三個專案代表了在 Godot 中取得 ECS 式架構（在其中一個案例中，甚至是 ECS 式效能）的三種結構上截然不同的策略：

1. godot-ecs —— 純 GDScript

  • 語言：100% GDScript，零外部依賴，不需要編譯 GDExtension（依 README 所述，「即插即用、易於除錯」）。
  • 架構：Entities 持有 Components（純資料）；Systems 有兩種模式——ECSSystem（單執行緒、有狀態的「Direct Mode」，用於一般遊戲邏輯）與 ECSParallel（多執行緒的「Scheduled Mode」，具備自動相依性分析，以決定哪些部分可以並行執行）。
  • Godot 整合方式：在由 ECSRunner 管理的獨立 ECSWorld 中執行，刻意與 Node/SceneTree 層級結構解耦。
  • 成熟度：積極維護中——328 顆星、25 個 release，最新版本 v3.1.0（2026 年 1 月），MIT 授權。
  • 但要注意：因為它是純 GDScript，其 systems 仍然透過 GDScript 解譯器執行。它提供了 ECS *組織上*的好處（乾淨的資料/邏輯分離、透過 ECSParallel 帶來的一些多執行緒能力），但沒有 DOTS 式的原生程式碼吞吐量——整套技術堆疊中沒有任何相當於 Burst 的編譯步驟。

2. flecs —— 原生效能核心

  • 語言：C（C99 API，零依賴），並在其上加了一層現代 C++17 API。
  • 架構：標準 ECS（entities/components/systems/queries），具備對快取友善的 archetype/SoA 儲存方式、無鎖（lockless）多核心排程器，並宣稱有能力達到每 frame 處理數百萬個 entities的規模。編譯時間不到 5 秒；8.5k GitHub 星數；已用於已上市的商業遊戲；在主要編譯器上有超過 13,000 項 CI 測試。
  • Godot 整合方式：原生完全沒有——flecs 與引擎無關。它有 C#、Rust、Zig、Lua 與 Clojure 的社群綁定，但要從 Godot 存取它，需要一個原生橋接方案。

3. Flecs-Godot 橋接方案（本次研究過程中發現，不在原始連結清單中）

  • Stagehand（VerdantInteractive）—— 一個將 flecs 帶入 Godot 的 GDExtension；透過 FlecsWorld node 將 ECS world 整合進 Godot 的場景層級結構/生命週期中。效能關鍵的程式碼以 C++ 撰寫，而 Godot 的腳本語言在其他地方仍可正常使用。
  • Glecs（GsLogiMaker/godot-glecs）—— 以 GDExtension 為基礎的 flecs Godot 綁定。
  • godot-turbo（callmefloof）—— 一個由 flecs 驅動的 ECS engine module（而非 GDExtension），目標為 Godot 4.6+。明確宣傳了一個 「Script Bridge」，可在 ECS entities 上執行 GDScript/C# 方法，並提供渲染橋接 nodes（InstancedRenderer3D、MultiMeshRenderer2D/3D、ComputeRenderer），將 ECS 驅動的資料直接連接到 Godot 的渲染/instancing 系統（這與先前 Shaders & GPU Instancing 條目有不錯的呼應）。
  • 三者共同的形態：ECS 模擬核心以原生 C/C++ 執行（flecs 本身）；GDScript/C# 只是作為其上的薄層腳本／橋接層。熱路徑邏輯存在於 C++ 中；GDScript 負責協調（orchestration）、面向玩法的掛鉤（hooks）以及設定。

GDScript vs C# 的問題——這才是真正的關鍵

這與本 repo 其他地方已經確立的模式相呼應（參見 DOTween/Tween 與 LINQ/Array）：GDScript 由於是解譯執行的，具有結構性的吞吐量限制，這個限制不會因為採用了 ECS 形狀的 API 就消失。

  • Unity 的 DOTS 之所以能達到那樣的效能，正是因為它在熱路徑上脫離了 C# 一般的執行模型——Burst 將受限的子集編譯成原生程式碼，Jobs 將其平行化。這兩者對 GDScript 都不存在。 沒有「給 GDScript 用的 Burst」這種東西。
  • Godot 的 C# 支援在架構上更接近 DOTS 所需要的東西（編譯執行、JIT 化），但 Godot 並沒有隨附官方 ECS 框架、與之綁定的相當於 Burst 的編譯器，或是給 C# 用的大量 entity job 系統——即使走 C# 這條路，你仍然得採用第三方方案。
  • godot-ecs（純 GDScript）提供了*組織上*的優勢，但沒有吞吐量上的優勢——GDScript 底層終究還是 GDScript。如果動機是程式碼結構而非原始的 entity 數量規模，這是合適的選擇。
  • flecs + 一個 Godot 橋接方案能提供與 DOTS 相當的原始效能——但熱邏輯必須用 C++ 撰寫，而不是 GDScript。所調查的三個橋接方案中，沒有一個能讓你用 GDScript 撰寫效能關鍵的模擬程式碼，同時仍取得 flecs 級別的吞吐量；在這三者之中，GDScript 的角色都僅止於膠合層。

在所調查的生態系中，不存在一種選項能在模擬邏輯本身以 GDScript 撰寫的情況下，提供與 Unity DOTS 相當的原始效能。 這種組合並不存在於所調查的生態系中——原因與 GDScript 沒有 Burst 編譯器的結構性理由相同：無論周圍的資料如何組織，一個解譯式語言都無法達到每 frame 數百萬個 entities 的吞吐量。

結論與建議

考量到目前設定的優先順序（「重點是使用 GDScript」），真正的決策在於這個專案採用 ECS 究竟是為了什麼——這才決定了哪條路徑（或是「完全跳過 ECS」）才是正確的：

  1. 如果目標是架構性的（將邏輯與 Node tree 解耦、更乾淨的 systems/資料分離、透過相依性排程的平行模式取得中等程度的多執行緒能力）——godot-ecs 是正確的選擇，而且是 GDScript 原生的，直接符合目前設定的偏好。除非 Unity 那邊特別依賴 DOTS 來取得原始 entity 數量的吞吐量，否則這很有可能就已足夠。
  2. 如果目標是與 DOTS 實際提供的效能對等的吞吐量（大規模模擬、人群、以 entity 表示的粒子達到 DOTS 級別的規模）——那就需要透過原生橋接方案使用 flecs（Stagehand/Glecs/godot-turbo），並將熱路徑邏輯以 C++ 撰寫，而不是 GDScript。「使用 GDScript」屆時只能適用於最外層的膠合／玩法層——這需要事先明確地做出取捨決定並被接受，而不是在遷移過程中才發現。
  3. 混合方案是可行的，而且可以說是最好的預設選擇：只針對真正需要大量 entity 吞吐量的特定子系統使用以 flecs 為基礎的橋接方案，其他部分則維持一般的 Godot Nodes/GDScript——這正好呼應了 Unity 自身的指引，即 ECS 與 GameObjects 是設計來共存的，而非全面取代。
  4. 不要預設就採用這三者中的任何一個。 這三者都是第三方方案，成熟度各不相同（godot-ecs：單一 addon 的 GDScript 專案，328 顆星；flecs 橋接方案：較新、較小型的 GDExtension 或 module 專案，*在 Godot 中*的生產環境實績尚未確認，儘管 flecs 本身在其他地方已有生產環境的實績）。請先確認專案實際的 DOTS 使用情況（見後續追蹤項目）——一個輕度使用 DOTS 的專案，可能完全不需要在 Godot 中採用任何 ECS 框架。

後續追蹤項目／待確認事項

  • 確認這個專案實際的 Unity DOTS/ECS 使用情況：哪些 systems 用到它、每個 frame 大致有多少 entities，以及它的存在是為了原始吞吐量還是僅僅是架構上的分離——這是決定哪種 Godot 方案（若有）適用的最大因素。
  • 如果原始吞吐量很重要，直接評估這三個 flecs-Godot 橋接方案（Stagehand、Glecs、godot-turbo）——它們是在本次研究過程中浮現的，而非原始請求中的項目，其成熟度／支援情況／API 穩定性在這一輪研究中並未深入驗證。
  • 確認團隊／build pipeline 是否能容忍熱路徑中出現 C++（技能組合、GDExtension 建置複雜度）——這決定了上述選項 2/3 是否現實可行，無論效能需求為何。
  • 如果選擇 godot-ecs（純 GDScript），應盡早針對專案實際的 entity 數量對其 ECSParallel 模式進行 benchmark——在解譯式語言中「一些多執行緒能力」與 Unity 的 Jobs+Burst 並不屬於同一效能等級，這一點應該被驗證，而不是被假設。
  • 檢查 Unity 專案中以 Burst 編譯的 systems 是否依賴 SIMD/向量化運算（這在 DOTS 程式碼中很常見)——如果是，這是純 GDScript 無法直接替代該程式碼的另一個原因，進一步強化了效能關鍵子系統需要原生橋接方案的必要性。
