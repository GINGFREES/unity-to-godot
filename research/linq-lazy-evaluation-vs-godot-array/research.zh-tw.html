Unity/C#：疊代器與 LINQ

  • 疊代器（yield return）：使用 yield return 的方法會被編譯器轉換成一個狀態機，回傳 IEnumerable／IEnumerator。執行方式是惰性（lazy）的：每次 foreach 的一個步驟，都會從上次暫停的地方繼續執行該方法，而不是一次把整個序列算完。這使得無限序列、分頁序列，以及把複雜的清單建構邏輯封裝在一個簡單且支援 foreach 的介面背後成為可能。
  • LINQ 延遲執行：LINQ 查詢變數在定義時並不會被求值，只有在被列舉（foreach、ToList()、.First() 等）時才會求值。這是建立在同一套疊代器機制之上的。
    • 惰性（Lazy）運算子（Where、Select……）每次拉取一個元素時才處理，隨著被消費而逐一求值。
    • 積極（Eager）運算子（OrderBy、ToList、ToArray，某些情況下的 Count）在回傳任何結果之前，必須先具象化／處理完整個來源（例如 OrderBy 必須看過所有元素才能排序）。
    • 延遲加上惰性的執行方式會「平均分攤負擔」，並可以避免在鏈式呼叫中具象化中介集合——對於大型集合上的鏈式運算來說，這是實際的效能優勢，並且能實現短路求值（例如 .Where(...).First() 可以在第一個符合項目後就停止，不需碰觸其餘的元素）。
    • 需注意的陷阱：由於求值是延遲的，重新列舉同一個查詢會重新執行一次——如果在定義查詢與列舉查詢之間，來源資料發生了變動，結果會反映當下的狀態，而不是定義當時的快照。這是一種常見且不易察覺的臭蟲來源。
  • Set 專用工具：HashSet<T>（真正的 Set 抽象資料型別，平均 O(1) 的成員檢查）以及 LINQ 中以 Set 為概念的運算子 Distinct()／Union()／Intersect()／Except()（以雜湊為基礎，同樣是延遲執行）。

Godot：Array

  • 用途：Godot 的通用序列容器。共有三種變體：未型別化的（Array，可存放任意 Variant，最有彈性但疊代速度最慢）、型別化的（Array[int]，速度較快且具型別安全）、以及 Packed 型別（PackedInt32Array、PackedVector2Array 等——針對特定型別最快且最省記憶體，但便利方法較少）。
  • 函式風格的方法：filter(method)、map(method)、reduce(method, accum)、any(method)、all(method)、find_custom(method, from)。
    • 這些方法全部都是積極（eager）的——每次呼叫都會立即完整處理／具象化出結果。GDScript 的 Array API 中並沒有延遲／惰性的鏈式組合：arr.filter(a).map(b) 會先完整建立出經過過濾的陣列，然後 map 才開始執行，這與 LINQ 的 .Where(a).Select(b) 鏈式呼叫不同，後者在每次疊代時會讓一個元素依序通過這兩個步驟。
    • any()／all() 內部確實會做短路求值（一旦得到確定的結果就停止）——但這只是單一方法層級的最佳化，並不是像 LINQ 那樣跨多個運算子組合而成的整條鏈式惰性求值。
  • 沒有原生的產生器／自訂疊代器語言關鍵字：Godot 4 以 await 取代了 Godot 3 的 yield()，但 await 是以信號（signal）為基礎的非同步機制，而非通用的惰性序列產生器。GDScript 中沒有直接對應到「用 yield return 撰寫方法，在每次迴圈步驟中惰性產生數值」的寫法。
  • 沒有原生的 Set 型別：官方文件中沒有提到把 Array 當作 Set 使用，也沒有專屬的 Set 類別。Array.has()／find() 都是 O(n) 的線性掃描。在 Godot 中，慣用的快速查找替代方案是把 Dictionary 當作 Set 使用（成員作為鍵，值則被忽略或設為 true），可提供平均 O(1) 的成員檢查——這才是更接近 HashSet<T> 的對應物，而不是 Array。
  • 其他需注意事項：在疊代過程中刪除元素是不受支援且行為不可預期的；陣列是以參照方式傳遞的（若需要獨立副本請使用 duplicate()）；GDScript 中的 const 陣列會自動成為唯讀。

比較／對應關係

C#/Unity Godot 備註
HashSet<T>，或 LINQ 的 .Distinct()／.Union()／.Intersect()／.Except() 把 Dictionary 當作 Set 使用（鍵 = 成員） 不是 Array——Array.has() 是 O(n)，Dictionary 查找則是 O(1)
List<T> + LINQ 的 .Where()／.Select() 鏈式呼叫 Array + .filter()／.map() 概念上相似，但 Godot 的版本是積極求值，而非惰性延遲求值
LINQ 延遲／惰性鏈式呼叫（低成本的組合方式、透過 .First() 短路求值） （沒有對應物） 必須手動改寫成明確的迴圈，搭配提前 break／return 才能達到相同的短路效益
自訂 yield return 疊代器（惰性／分頁／無限序列） （沒有語言層級的對應物） 需要手刻一個具備手動狀態管理的類別；請確認目前 Godot 版本對自訂疊代器協定的支援情況（見後續追蹤事項）
List<int>、Vector2[] 等（型別化的泛型集合） 型別化的 Array[int] 或 Packed 陣列（PackedInt32Array、PackedVector2Array） 比起純粹未型別化的 Array，優先選用這些型別以取得更接近的效能與型別安全對等性

回答「在 Godot/GDScript 中處理 Set，哪種方式比較好」

這取決於此處所指的「Set」是哪一種意涵：

如果「Set」指的是 Set 抽象資料型別（唯一性、快速的成員檢查）：

請使用 Dictionary（把鍵當作成員，值則忽略不用）——而不是 Array。這是 C# 的 HashSet<T> 或 LINQ 的 Distinct／Union／Intersect／Except 用法，直接且正確的遷移目標。Array 本質上是清單；若用它來實現 Set 語意，代表每次成員檢查都要承擔 O(n) 的成本。

如果「Set」指的是一般性的項目集合／資料集合（較有可能的解讀）：

Array 的 filter()／map()／reduce() 對於一次性或非熱路徑的邏輯來說，是 LINQ 的 Where()／Select()／Aggregate() 一個合理且可讀性佳的替代方案。但它們是積極求值的，因此：

  • 對大型集合進行長串運算子鏈式呼叫時，每一步都會完整具象化，而 LINQ 原本可能保持惰性，避免產生中介配置。
  • 任何依賴短路求值（一旦找到足夠的結果就提前停止）的程式碼都沒有現成的對應物，需要改寫成搭配 break 的明確迴圈。
  • 任何依賴延遲執行「重新列舉即可反映當下狀態」行為的程式碼都需要重新設計；Godot 並沒有可供延遲求值的查詢物件概念。
  • 對於熱路徑／每影格執行的程式碼而言，LINQ 鏈式呼叫與 Array 的函式風格方法，相較於手寫迴圈都帶有實質的每次呼叫額外開銷（委派／Callable 呼叫成本）——這項取捨在兩個引擎之間是對稱的，並不構成偏好其中一方的理由。在效能關鍵的程式碼中，無論在哪一側，都應優先選用明確的迴圈。

結論與建議

  1. 不要把 Array 當成 HashSet<T> 的替代品。如果要遷移原本使用 HashSet<T> 或 LINQ 的 Set 運算子來實現真正 Set 語意的程式碼，應該改用 Godot 中「把 Dictionary 當作 Set」的做法——這是唯一一個無論考不考慮積極／惰性求值，Array 都是錯誤工具的情況。
  2. 對於一般的「LINQ 操作清單」程式碼，Array 的函式風格方法在可讀性上是可以接受的，但要預先評估行為上的差異：沒有延遲執行、沒有跨鏈式呼叫的短路求值、也沒有「重新列舉即反映變動」的語意。任何依賴這些 LINQ 特有行為的程式碼，都需要改寫成明確的迴圈，而不能只是機械式地把 .Where() 換成 .filter()。
  3. 優先選用型別化／Packed 陣列（Array[T] ／ PackedXArray），而不是未型別化的 Array，當原本來源使用的是 List<T>／型別化陣列時——可取得更接近的效能與型別安全對等性。
  4. 自訂的惰性產生器（以 yield return 為基礎）在 GDScript 中沒有直接對應物——請規劃以明確狀態手刻實作這類邏輯，並且在假設能夠乾淨移植之前，先確認目前 Godot 版本提供了哪些自訂疊代器支援（若有的話）。
  5. 在熱路徑中，應同時避免 LINQ 鏈式呼叫與 Array 的函式風格方法——無論在哪一側撰寫程式碼，都改用明確的迴圈。這是雙方共通的最佳實務，而不是偏袒任何一個生態系的論點。
