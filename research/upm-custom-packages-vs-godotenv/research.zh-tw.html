UPM 自訂套件（Unity）與 GodotEnv（Godot）比較

官方資料來源

  • Unity Custom Packages（UPM）：https://docs.unity3d.com/6000.4/Documentation/Manual/CustomPackages.html
  • GodotEnv（Chickensoft）：https://github.com/chickensoft-games/godotenv

Unity：自訂套件（UPM）

  • 目的：以可重複使用、可分享的單元擴充編輯器／專案功能——可在內部跨專案／團隊使用，或對外公開發佈。
  • 建立方式：Package Manager 的 (+) > Create package 會產生一個初始資料夾與 manifest。
  • Manifest（package.json）：包含名稱、版本、描述、相依性——隨著套件成熟會手動擴充。
  • 程式碼組織：.asmdef assembly definition 檔案會將套件的 C# 程式碼隔離成獨立的編譯組件（compiled assembly）。
  • 套件來源：embedded（位於專案的 Packages/ 資料夾內）、local（路徑參照），或 git（在專案的 manifest 中以 git URL 參照）。
  • 版本管理／發佈：採用語意化版本（semantic versioning）；對於可分享的套件，建議／預期附上 changelog、授權／法律文件、文件、測試與範例。
  • 參照方式：由使用端專案自身的 manifest.json 相依性清單中參照（透過 registry 名稱、git URL，或檔案路徑）。

Godot：GodotEnv

  • 目的：一個第三方 CLI 工具（MIT 授權，由 Chickensoft 開發），用來解決 UPM 在 Unity 中合併為一體的兩個獨立問題——Godot 引擎版本管理與 addon 相依性管理——因為 Godot 本身並未內建這兩者中的任何一個。
  • 版本管理：可在 Windows／macOS／Linux 上安裝／使用／解除安裝特定的 Godot 版本（3.x 與 4.x，.NET 或 standard 版本）；會維護一個指向目前使用版本的 symlink，並設定 GODOT 環境變數。
  • Addon 管理：相依性會宣告在專案根目錄的 addons.json／addons.jsonc 中。每個項目會指定：
    • addon 名稱（必須與其安裝資料夾名稱相符）
    • 來源 URL（git repo、本機路徑、symlink，或 zip）
    • 來源類型（remote／local／symlink／zip）
    • git checkout 參照（分支或標籤）
    • 來源中選用的子資料夾路徑
  • 傳遞性相依（Transitive dependencies）：一個 addon 本身也可以附帶宣告自身相依性的 addons.json；GodotEnv 會將這些相依性攤平解析（resolves these flatly）。
  • 安全性：在重新安裝前會偵測已安裝 addon 是否有本機修改（不會在使用者不知情的情況下覆蓋本機修改）。
  • 核心指令：godotenv godot install/use/uninstall [version]、godotenv addons init、godotenv addons install、godotenv config list/set。

比較／對應關係

Unity（UPM） Godot（GodotEnv）
每個套件一個 package.json manifest 每個 addon 一個 addons.json 項目
Registry／git URL／本機路徑來源 git repo／本機路徑／symlink／zip 來源（無 registry 概念）
.asmdef 編譯組件隔離 沒有針對各個 addon 的編譯組件隔離——Godot 的 C# 通常是單一的專案層級組件（project-wide assembly）
內建於編輯器中（Package Manager 視窗、Unity 自身的 registry、scoped registries） 第三方 CLI 工具，並未內建於 Godot 編輯器中
由 UPM 解析巢狀套件相依性 由 GodotEnv 將巢狀 addons.json 攤平解析
關鍵落差：UPM 是官方工具，具備編輯器內的 UI 以及真正的套件 registry（包括 scoped／私有 registries）。Godot 沒有官方對應的等效工具——其內建的 Asset Library 只是一個編輯器內的外掛瀏覽器，而不是以 manifest 驅動、可鎖定版本、可進行傳遞性解析的相依性管理系統。GodotEnv（社群工具）是功能上最接近的替代方案，但它僅支援 git／本機／zip 為基礎的來源——並沒有私有 registry 的概念可以將私有的 UPM scoped registry 遷移過去。

回答隱含的問題

1. 是否有官方的 Godot 版本可對應 UPM？

沒有。Godot 內建的 Asset Library 涵蓋了手動、編輯器內的外掛探索／安裝，但不具備可重現的相依性管理（沒有 git-ref 鎖定、沒有傳遞性解析、沒有 CLI／CI 腳本化支援）。GodotEnv 是既有的社群替代方案，也是最接近 UPM 功能的選項。

2. 這個專案能否直接使用 Godot 內建的 Asset Library 來取代 GodotEnv？

只適用於簡單、手動管理的情境。它缺乏可重現的版本鎖定、傳遞性解析、本機修改偵測，以及對 CI 友善的腳本化安裝——而這些都是以 UPM 為基礎的自訂套件工作流程很可能已經仰賴的功能。GodotEnv（或是自行打造的、等效於 addons.json 的方案）會是更貼近原本需求的替代方案。

3. 遷移的形式

每一個既有的 UPM 自訂套件都會變成一個 Godot addon 資料夾：

  • 如果它是僅供編輯器使用的工具 → 需要一個 plugin.cfg，以及 _enter_tree()／_exit_tree() 外掛生命週期函式。
  • 如果它是僅供執行期使用的函式庫 → 只需要一個包含 scripts／resources 的資料夾，不需要 plugin.cfg，並透過 autoload／preload 接入。

接著每個 addon 都會被宣告為一筆 addons.json 項目，指向其來源目前所在的位置（可能是其自身的 git repo，或是在遷移過渡期間所使用的本機路徑）。

待辦事項／之後需要驗證的事項

  • 這個專案的自訂 UPM 套件是僅供編輯器使用的工具、執行期函式庫，還是兩者混合？這將決定 Godot 端的 addons 是否需要 plugin.cfg。
  • 這些套件目前是否託管在私有／scoped 的 UPM registry 上？如果是，就需要為每個套件規劃一套 git 託管方案，因為 GodotEnv 沒有私有 registry 的概念——只支援 git／本機／zip 來源。
  • 是否有任何 C# 自訂套件依賴 .asmdef 所驅動的組件隔離，用於建置時的分離或存取控制？Godot 通常採用單一的專案層級 C# 組件，這可能需要重新思考該邊界的設計方式。
  • 在將 GodotEnv 採用為團隊標準工具之前，需確認其目前的版本／成熟度，以及任何已知問題（例如 Windows 上的 symlink 需求）。
